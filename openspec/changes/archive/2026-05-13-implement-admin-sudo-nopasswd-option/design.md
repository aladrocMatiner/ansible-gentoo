# Design: implement-admin-sudo-nopasswd-option

## Overview

The project keeps real installation access conservative while making disposable libvirt validation VMs easier to debug. The shared users role owns sudo policy for both OpenRC and systemd installs, so the option is implemented once in `common/users` and validated once in `common/final_checks`.

## Variables

- `ADMIN_SUDO_NOPASSWD`: operator-facing Makefile variable.
- `admin_sudo_nopasswd`: Ansible variable passed to users and validation roles.
- `VM_E2E_ADMIN_SUDO_NOPASSWD`: libvirt E2E harness default, set to `yes`.

Rules:

- `ADMIN_SUDO_NOPASSWD` accepts only `yes` or `no`.
- Unset `ADMIN_SUDO_NOPASSWD` is treated as `no` in normal install and configure-users workflows.
- `vm-e2e-install` exports `ADMIN_SUDO_NOPASSWD=${ADMIN_SUDO_NOPASSWD:-$VM_E2E_ADMIN_SUDO_NOPASSWD}` before invoking the full install.
- Operators can override disposable VM behavior with `ADMIN_SUDO_NOPASSWD=no`.

## Sudoers Policy

The target sudoers drop-in remains `/etc/sudoers.d/gentoo-ai-installer-admin`.

When disabled:

```sudoers
%wheel ALL=(ALL:ALL) ALL
```

When enabled:

```sudoers
%wheel ALL=(ALL:ALL) NOPASSWD: ALL
```

All configured admin groups receive the same mode so group behavior stays consistent.

## Validation

`config-check` validates the variable shape and prints whether passwordless sudo is enabled without printing secrets.

`common/final_checks` validates:

- sudoers syntax with `visudo`,
- the admin user and group membership,
- `NOPASSWD: ALL` is present when requested,
- `NOPASSWD:` is absent when not requested.

First-boot validation runs `sudo -n true` only when passwordless sudo was requested. That proves the SSH admin user can elevate without an interactive password in disposable test VMs.

## Safety

Passwordless sudo is high-risk persistent target behavior. It is acceptable by default only in the disposable libvirt E2E path because those VMs are project-owned test artifacts. Real hardware and remote network installs retain the safer default.

No plaintext passwords or password hashes are generated by this change. Operators who want password-requiring sudo must provide `ADMIN_PASSWORD_HASH_FILE` if they need interactive sudo after first boot.
