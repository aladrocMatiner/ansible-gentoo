# Design: implement-install-state-and-resume-checkpoints

## State Location

State must be project-local and ignored from git when it contains run-specific data.

Recommended layout:

```text
var/state/current-install.json
logs/install-runs/<run-id>/
  state.json
  events.jsonl
  summaries/
```

The exact layout may change during implementation, but it must remain under project-controlled paths and must not write to arbitrary host locations.

## Run Identity

Each install attempt should have a run id. The run id must be generated by the Makefile/script/Ansible workflow and printed at the start of an operation.

The run state should record non-secret values:

- profile/init system,
- filesystem,
- boot mode,
- target mount path,
- selected install disk path,
- disk model, serial when available, and size,
- partition UUIDs and filesystem UUIDs when they exist,
- completed phase checkpoints,
- validation timestamps,
- log artifact paths.

## Resume Rules

Resume is allowed only when current system facts match the recorded facts for the last completed checkpoint.

Resume must fail closed when:

- disk identity differs,
- expected partitions or UUIDs differ,
- mount state differs from the checkpoint,
- selected `PROFILE` or `FILESYSTEM` differs,
- a destructive confirmation is missing for a destructive next step,
- required logs or checkpoint files are missing,
- state contains fields that look like secrets.

Resume must not skip shared destructive safety gates.

## Makefile Integration

All operator-facing state actions must be make targets. Planned targets may include:

- `make install-state`
- `make install-resume-plan`
- `make install-run-clean`

Implementation changes must document any actual target names in `README.md` or `docs/`.

## Documentation

Document state files, recovery behavior, failure modes, and what is safe to delete. Do not document secrets or include local credentials in examples.
